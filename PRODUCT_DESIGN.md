# IKR 产品设计方案

---

## 一、整体架构

### 用户旅程全景

```
内容收集          知识处理           价值兑现
─────────         ─────────          ─────────
微信转发    →    自动消化           →   提问
浏览器插件  →    知识卡片生成       →   获得多方案
（未来）         源头资料补充           决策更有底气
                 语义关联
                 静默复习调度
```

### 平台分工

| 平台 | 职责 | 理由 |
|------|------|------|
| 微信服务号 | 内容入口 + 日常对话（多轮，轻量） | 用户在哪里消费内容，就在哪里收录和提问 |
| 微信小程序 | 深度交互：完整对话历史、知识库、图谱 | 微信生态内承接深度场景，无跳出割裂感 |
| Web App | 保底兜底 | 小程序审核风险的备用方案 |
| 浏览器插件（Phase 3）| 扩展内容来源 | 覆盖非微信内容 |

**核心设计原则：全程留在微信生态内，服务号处理日常，小程序承接深度。**

## 二、核心功能设计

### 功能一：内容收录（微信端）

**用户操作流程：**
1. 在微信看到文章 → 分享给 IKR 服务号
2. 服务号回复确认："已收录，正在消化…"
3. 消化完成后推送通知："《XXX》已消化完毕，提炼了 5 个知识点"（可点击查看）

**支持的内容类型（分阶段）：**
- Phase 0：微信公众号文章
- Phase 1：任意网页链接、视频链接（B站、YouTube）
- Phase 3：PDF、浏览器插件一键收录

**关键设计决策：**
- 收录确认要快（< 3 秒），消化可以异步
- 通知要轻，不强迫用户点开，看到标题知道"存进去了"就够

### 功能二：自动消化（后台）

#### 2.1 内容分类体系

消化前先识别内容类型，不同类型使用不同的提取逻辑。分类按**信息结构**而非内容形式，共 4 个主类型：

| 类型 | 特征 | 典型来源 |
|------|------|---------|
| **论点型** | 作者在论证一个主张，有观点有依据 | 评论文章、观点类公众号 |
| **操作型** | 描述如何做某件事，有步骤有条件 | 教程、方法论、操作指南 |
| **事实型** | 陈述数据、定义、客观规律 | 行业报告、研究数据、概念解释 |
| **叙事型** | 讲案例、故事、复盘 | 公司案例、人物故事、经验总结 |

**分类规则：**
- 一篇内容可以同时包含多种类型，AI 识别主类型 + 次要类型
- 主类型决定整体摘要结构，次要类型对应段落单独提取
- 事实型卡片附加时效性标注（提取发布时间，超过 2 年标注"注意时效"）

#### 2.2 处理流程

```
原始内容
    ↓
内容清洗（去广告、去无关元素）
    ↓
内容分类识别
  - 识别主类型（论点/操作/事实/叙事）
  - 识别次要类型（按段落）
    ↓
分类型摘要生成（不同类型不同结构）
    ↓
分类型知识卡片提取（5-10 张）
  - 每张卡片携带类型标注
  - 事实型附加时效性
    ↓
语义向量化 → 存入向量数据库
    ↓
与已有知识库做相似度匹配 → 建立关联
    ↓
源头资料检索（独立参考层）
  - 搜索相关学术文献、原始报告、经典资料
  - 不混入个人知识库，单独标注
```

#### 2.3 各类型提取结构

**论点型提取：**
```
【主张】用户增长的北极星指标应该反映核心用户价值
【依据】DAU 等虚荣指标无法反映用户真实获得的价值
【适用条件】适用于有明确核心用户行为的产品
【局限】作者承认：早期产品难以定义核心行为
【来源】《增长黑客》· 论点型 · 2024-03-20
```

**操作型提取：**
```
【做法】通过留存曲线分析找到流失关键节点
【前提】已有至少 3 个月的用户行为数据
【步骤】1. 按注册时间分组 → 2. 画留存曲线 → 3. 找断崖点
【判断标准】能定位到具体的流失时间窗口
【来源】《SaaS 增长手册》· 操作型 · 2024-03-20
```

**事实型提取：**
```
【事实】SaaS 产品平均 NRR 超过 120% 才能支撑高估值
【来源】A16Z 2023 年 SaaS 基准报告
【样本范围】北美 200+ SaaS 公司
【时效】2023 年数据，注意时效
【类型】事实型 · 有时效风险
```

**叙事型提取：**
```
【案例】Airbnb 通过专业摄影服务提升房源转化率
【背景】早期房源图片质量差，转化率低
【决策】创始人亲自上门为房东拍照
【结果】试点城市收入翻倍
【可迁移规律】早期不规模化地做事，找到真正的转化瓶颈
【来源】《创业维艰》· 叙事型 · 2024-03-20
```

### 功能三：问题解答（服务号 + 小程序）

#### 3.1 可实操回答的核心设计

**IKR 回答的本质要求：用户看完之后，知道明天早上第一步做什么。**

不能停在框架层面（"重新定义北极星指标"是概念，不是操作）。
可实操 = 前提条件 + 具体步骤 + 判断标准。

**实现可实操的三层机制，按优先级依次触发：**

**第一层：从知识库推断用户背景（无感，自动）**

用户积累的内容本身透露了他的行业、角色、关注点。
每次提问前，系统自动分析知识库画像：

```
知识库画像推断示例：
- 用户收录了大量 SaaS、B端、企业服务相关内容
  → 推断：B端产品从业者
- 用户收录了增长、留存、DAU 相关内容
  → 推断：关注用户增长的产品/运营角色
- 用户近期收录了融资、PMF 相关内容
  → 推断：处于早期创业阶段

结合画像，同一个问题给出不同的实操建议：
- B端 SaaS 增长停滞 → 优先建议客户成功和扩展收入
- C端 App 增长停滞 → 优先建议留存漏斗和渠道效率
```

**第二层：每个方案内置实操结构（分场景，不追问）**

每个方案固定包含四个要素：

```
▌ 方案名称（推荐 / 备选）
  【适用前提】满足什么条件时选这个方案
  【具体步骤】第一步做什么 → 第二步做什么（2-4步，可执行）
  【判断标准】怎么知道做对了 / 做完了
  【知识来源】来自你的知识库 / 通用知识
```

**第三层：问题模糊时，问一个最关键的澄清问题（有摩擦，谨慎触发）**

仅当问题模糊到无法区分实操路径时才触发，且只问一个问题：

```
触发条件示例：
- "我应该怎么做决策" → 太宽泛，需要澄清场景
- "团队管理有什么建议" → 无法判断团队规模和问题类型

触发格式：
IKR → 我需要了解一个关键信息才能给出具体建议：
      你现在的团队规模大概是多少人？
      A. 5人以下  B. 5-20人  C. 20人以上
```

只问一个问题，给出选项，降低回答成本。

---

#### 3.2 回答格式示例

**完整回答示例（知识库有相关积累）：**

```
你问：产品增长停滞了，有哪些方向可以突破？

[IKR 内部推断：用户知识库显示 B端 SaaS 背景，
 近期关注增长和留存，推测处于成长期]

基于你的知识库和背景，有 3 个方向：

▌ 方向一：修复核心用户的留存漏斗（推荐）
  来自《用户行为分析》《SaaS 增长手册》

  【适用前提】新用户 30 天留存率低于 40%
  【第一步】按注册时间分组，画出各批次的留存曲线
  【第二步】找到留存曲线断崖的时间点（通常在第 3-7 天）
  【第三步】访谈在该时间点流失的用户，找共同原因
  【第四步】针对原因设计一个 onboarding 干预实验
  【判断标准】新用户 7 日留存率提升 10% 以上

▌ 方向二：提升现有客户的扩展收入
  来自《增长黑客》

  【适用前提】留存尚可（>60%），但新客获取成本上升
  【第一步】统计现有客户中有扩展购买行为的比例和触发条件
  【第二步】设计针对高价值客户的主动扩展流程
  【判断标准】NRR（净收入留存率）超过 110%

▌ 方向三：重新定义北极星指标
  来自通用知识

  【适用前提】团队对"增长"的定义模糊，各部门目标不一致
  【第一步】列出现在追踪的所有指标，找出与用户核心价值
           最相关的那一个
  【第二步】以该指标为核心，重新对齐团队 OKR
  【判断标准】全团队能用一句话说清楚北极星指标是什么

延伸参考：A16Z《Growth Accounting Framework》
```

#### 3.3 深度思考模式

**问题背景：** 长期只给答案，用户会失去独立思考能力，与"提升认知"的底层价值矛盾。

**解法：两种模式，用户主动选择。**

每次给出决策方案后，提供切换入口：

```
IKR 回答完方案后：

━━━━━━━━━━━━━━━━━━━━━━━━
[用这个方案]   [我想深入理解这个问题]
━━━━━━━━━━━━━━━━━━━━━━━━
```

**决策模式（默认）：** 给可实操方案，现在就能用。

**深度思考模式（用户主动触发）：**

切换后 IKR 不再给答案，而是：

1. **给出思考框架**：优先调用用户知识库里已有的相关框架
2. **苏格拉底式追问**：引导用户自己推导，而非灌输结论
3. **补充盲点**：用户推导完成后，IKR 只补充遗漏的关键视角

```
用户选择"深入理解这个问题"后：

IKR → 好，我们来深入想这个问题。

      你上周读过《用户行为分析》里提到的
      "留存漏斗"框架，试着用它来分析：

      你认为用户在哪个阶段开始流失的？
      A. 注册后 3 天内
      B. 第一次核心操作之后
      C. 付费转化节点

用户 → B，第一次核心操作之后

IKR → 有意思。那你觉得为什么完成了核心操作
      之后还会流失？是操作体验的问题，还是
      用户没有感受到预期的价值？

用户 → 我觉得是没感受到价值

IKR → 那"价值"对你的用户来说具体是什么？
      他们使用这个产品想解决什么问题？

（引导用户自己推导出根本原因）

用户推导完成后：

IKR → 你的分析很准确。补充一个你可能没考虑到的
      视角：根据你收录的《SaaS 增长手册》，
      这类问题通常还有一个隐藏原因是...
```

**关键设计原则：**
- 深度思考模式里调用的框架，优先来自用户自己的知识库——"你读过的 X 框架"比"有个框架叫 X"更有说服力，也强化了积累的价值感
- IKR 在这个模式里只问问题，不给结论，直到用户自己推导出来
- 最后的"盲点补充"是点睛之笔，让用户感受到 IKR 的价值，而不是"我自己想的，要你何用"

#### 3.4 服务号交互流程

用户直接在服务号对话框提问，后端以 openid 维护会话上下文，支持多轮追问。

**服务号 → 小程序的自然跳转时机：**
- 用户追问超过 3 轮："这个话题比较深，在小程序里继续会更清晰 →"
- 用户问"我之前收录过什么"："查看你的知识库 →"
- 用户需要查看完整对话历史："在小程序里查看 →"

**小程序（深度对话）：**

承接从服务号跳转过来的对话，保留完整上下文，支持：
- 无限轮次多轮对话
- 对话历史查看
- 知识库浏览和搜索
- 知识图谱（Phase 2）

**冷启动状态（知识库 < 5 篇）：**

知识库画像不足时，第一层推断降级为通用假设，第二层实操结构不变，回答仍然可操作，但适用前提会更宽泛。同时提示：

```
（你的知识库还在积累中，目前基于通用知识给出建议。
  收录内容越多，建议会越贴合你的实际情况。）
```

### 功能四：知识库（小程序）

**设计原则：供查看，不是供管理。**

用户不需要在这里做任何事，这里是"看见自己积累了什么"的地方。

**主要视图：**

**① 时间流视图（默认）**
按收录时间倒序，每条显示：
- 文章标题 + 来源
- 3 条核心论点（折叠展开）
- 提炼的知识卡片数量
- 与其他内容的关联数

**② 知识图谱视图（Phase 2）**
- 节点 = 知识卡片
- 连线 = 语义关联
- 聚类 = 自动识别的知识领域
- 让用户看到："我在哪些领域积累最深，哪些是盲区"

**③ 搜索**
- 支持自然语言搜索（"关于增长的内容"）
- 不是关键词匹配，是语义搜索
- 搜索结果直接可以追问

---

### 功能五：知识留存（后台静默）

**用户完全感知不到这个机制的存在。**

后台做的事：
- 记录每个知识卡片最后一次被"激活"的时间（被提问调取 = 激活）
- 根据遗忘曲线计算每张卡片的"遗忘风险"
- 提问时，在语义相关的卡片中，优先选择遗忘风险高的卡片来构建回答
- 效果：用户越常提问，知识库的留存率越高，回答质量越好

用户感知到的只是："IKR 的回答越来越准了"

## 五、数据模型（概念层）

```
用户
 └── 知识条目（文章）
      ├── 元信息（标题、来源、收录时间）
      ├── 结构化摘要
      ├── 知识卡片[]
      │    ├── 知识点内容
      │    ├── 原文出处
      │    ├── 向量嵌入
      │    ├── 最后激活时间
      │    └── 遗忘风险分值
      ├── 语义关联[]（→ 其他知识条目）
      └── 源头参考[]（独立，不混入知识库）

对话记录
 ├── 问题
 ├── 回答（含方案列表）
 └── 调用的知识卡片[]（用于更新激活时间）
```

## 六、MVP 范围界定

### Phase 0 必须有

| 功能 | 说明 |
|------|------|
| 微信服务号接收文章 | 核心入口 |
| 文章自动消化 | 摘要 + 知识卡片 |
| Web 端提问 | 最核心的价值兑现 |
| 分层回答 | 你的知识库 vs 通用知识，清晰区分 |
| 冷启动体验 | 知识库为空时通用 AI 兜底 + 进度提示 |

### Phase 0 不做

| 功能 | 原因 |
|------|------|
| 知识图谱可视化 | 需要足够多内容才有意义 |
| 源头资料自动补充 | 核心闭环验证后再加 |
| 视频内容支持 | 复杂度高，先验证文章场景 |
| 间隔复习调度 | 先有足够知识库再优化留存 |
| 浏览器插件 | Phase 3 |

## 七、成功指标

### Phase 0 验证指标

| 指标 | 目标 | 说明 |
|------|------|------|
| 同日转发+提问率 | > 40% | 核心 PMF 验证：时间差真的压缩了 |
| 用户主观反馈 | "比 ChatGPT 更有用" | 个性化价值被感知 |
| 次周留存 | > 50% | 用户愿意回来 |

### Phase 1 健康指标

| 指标 | 目标 |
|------|------|
| 月留存率 | > 65% |
| 付费转化率 | > 15% |
| 人均月提问次数 | > 10 次 |
| 人均月收录篇数 | > 15 篇 |
